## 四、关键前提（踩过的坑，已解决）

| 阶段 | 问题 | 解决 |
|------|------|------|
| ① 连接器授权 | 连接器 OAuth token 失效（401） | 重新授权 GitHub 连接器 |
| ② 仓库写权限 | `ouyang520` 对仓库只有 `pull` | 仓库 owner 邀请 `ouyang520` 为 Write 协作者并接受 |
| ③ git 推送凭证 | 连接器返回的是 GitHub App token（`ghu_`），**只能读 API，不能 git 推送**（403 `Resource not accessible by integration`） | 改用 **PAT（个人访问令牌 `ghp_`）** 作为 git 推送凭证，写权限生效 |

> **结论**：WorkBuddy 的 GitHub 连接器（GitHub App）只提供只读能力；真正的写操作（git push / 自动部署）必须用一个**对该仓库有写权限的 PAT**。这把 PAT 即团队统一的提交身份凭证。

## 五、日常协同流程（极简模式，已验证可行）

1. 不懂技术的同事在 WorkBuddy 提需求 → AI 改代码

## 六、安全提醒

- PAT 具有仓库写权限，请妥善保管；如在聊天/日志中暴露，立即到 GitHub 重新生成（revoke 旧的）。
- 本测试中使用的 PAT 仅本次会话内存使用，未写入任何文件、未持久化。
- 多人协同时，PAT 由管理员统一配置到各 WorkBuddy 工作区的 git 凭据，不向个人分发明文。
