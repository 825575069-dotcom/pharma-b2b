# pharma-b2b 极简协同 + 自动部署（无审阅模式）

> 目标：**非技术同事用 WorkBuddy 开发后，AI 直接提交并部署上线**，无需 PR、无需人工审阅 / 合并。
> 配套：`DEPLOY.md`、`README.md`、`.github/workflows/deploy.yml`（部署入口）。

---

## 1. 极简原则

1. 同事只提需求，AI 改完代码**直接 push `main` → 自动部署**，全程无 PR、无审阅人。
2. 所有提交统一企业 Git 身份（bot 账号）。
3. push `main` 即触发 `deploy.yml` 自动构建并部署到服务器。

---

## 2. 一步到位的流程

```
非技术同事在 WorkBuddy 说需求
      ↓
WorkBuddy（AI）改代码
      ↓
AI 先 git pull --rebase origin main   ← 关键：拉最新、自动合并，防互相覆盖
      ↓
AI 提交（统一身份，message 标真实作者）
      ↓
AI push origin main
      ↓
GitHub Actions deploy.yml 自动构建并部署到服务器
```

同事全程无需懂 git / GitHub，说完需求等"已上线"即可。

---

## 3. 统一 Git 身份（管理员一次配置）

- 用一个企业 GitHub 账号（如 `pharma-b2b-bot`）+ 一把 repo 写权限的 PAT，作为所有提交的 committer。
- PAT 存企业密钥库 / WorkBuddy 连接器，不发给个人、不进群、不写仓库。
- 每个 WorkBuddy 会话配置：
  ```bash
  git config user.name  "pharma-b2b-bot"
  git config user.email "bot@your-company.com"
  git remote set-url origin https://<PAT>@github.com/825575069-dotcom/pharma-b2b.git
  ```
- commit message 模板（可追溯，非审阅）：
  ```
  <type>: <简述>（开发者：<真实姓名> 经 WorkBuddy 生成）
  ```

---

## 4. 关键：push 前必须 `pull --rebase`

多人直接 push `main` 会互相覆盖 / 冲突，这是"无审阅即时部署"唯一绕不开的技术点。AI 在 push 前自动执行：

```bash
git pull --rebase origin main
git push origin main
```

- **无冲突**：自动变基后推送，部署照常进行。
- **有冲突**（同一文件同一区域两人同改）：AI 尝试自动解决；解决不了则提示用户（极少发生，且非"审阅"，只是告知）。

---

## 5. 给非技术同事的操作（只需 3 步）

1. 打开 WorkBuddy，进入 pharma-b2b 会话。
2. 用大白话描述需求，例如：
   - "首页加一个轮播图"
   - "把管理后台标题改成『医药 B2B 平台』"
3. 说"**提交并部署**"，等 AI 回复"已部署完成"。

就这三步。不需要知道分支、PR、合并是什么。

> 可选：若你连"提交并部署"都不想说，可让 AI 改完**自动**部署（全自动模式）。建议至少保留一句"确认部署"防手滑，但这不属于审阅。

---

## 6. 风险与轻量回滚（不增加审阅负担）

无审阅闸门 = 有问题的代码会直接上线。保留**快速回滚**能力即可：

- 部署出问题：
  ```bash
  git revert <问题commit>          # 生成一个反向提交
  git push origin main             # 重新触发部署，回退到上一版
  ```
- 或给 `deploy.yml` 增加一键回滚步骤（保留上一个可部署版本）。
- 统一身份的 message 必须标真实作者，出问题好定位。

---

## 6.1 常见疑问：每人都要登录同一个 GitHub 账号吗？

**不需要。** 这里有两个身份，彼此解耦：

| 身份 | 用途 | 是否需统一 |
|------|------|-----------|
| WorkBuddy 登录（企业账号） | 使用 AI 对话、企业鉴权 | 各自独立，每人用自己的账号 |
| Git 提交身份（共享 PAT / bot 账号） | 代码 commit / push | 统一为一个企业 bot 账号 |

正确做法：
- 每人用自己的 WorkBuddy 企业账号登录（各自用 AI，互不影响）。
- 统一提交身份靠一把**共享 PAT**（由企业 bot 账号生成），配置到每个工作区的 git remote：
  ```bash
  git remote set-url origin https://<PAT>@github.com/825575069-dotcom/pharma-b2b.git
  ```
- 结果：GitHub 上所有提交都显示同一个 bot committer（统一），但每个人用的是自己的 WorkBuddy 登录。

> 若企业连接器本身已是「统一 GitHub 身份」授权，则连 PAT 都无需每人配置——连接器即统一身份。无论哪种，都**不要求每人去 GitHub 登录同一个账号**。
